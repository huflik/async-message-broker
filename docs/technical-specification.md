
# Техническое задание

## на разработку асинхронного брокера сообщений с гарантированной доставкой и персистентным хранилищем на основе библиотек Boost.Asio и ZeroMQ

---

## 1. Введение

### 1.1. Наименование работы
**«Асинхронный брокер сообщений с гарантированной доставкой и персистентным хранилищем на основе библиотек Boost.Asio и ZeroMQ»**

### 1.2. Краткое описание
Разрабатывается отдельный сетевой сервис (брокер), обеспечивающий надёжный асинхронный обмен сообщениями между распределёнными процессами. Брокер гарантирует доставку сообщений даже при временной недоступности получателя за счёт сохранения на диск, поддерживает двухстороннюю связь (request-reply) с корреляцией сообщений и предоставляет клиентам асинхронный интерфейс через ZeroMQ.

---

## 2. Назначение и цели разработки

### 2.1. Назначение
Обеспечение надёжного асинхронного обмена сообщениями между распределёнными компонентами (процессами, микросервисами) с гарантированной доставкой и поддержкой обратной связи.

### 2.2. Цели разработки
1. Реализовать брокер как самостоятельный процесс (не встраиваемая библиотека)
2. Обеспечить гарантированную доставку сообщений за счёт персистентного хранения на диске
3. Реализовать двухстороннюю связь с корреляцией запросов и ответов
4. Обеспечить отложенную доставку сообщений для офлайн-получателей
5. Реализовать асинхронную обработку всех операций на базе Boost.Asio
6. Использовать ZeroMQ для транспорта и маршрутизации
7. Предоставить клиентам API для регистрации, отправки сообщений и получения ответов

---

## 3. Функциональные требования

### 3.1. Регистрация клиентов
- Клиент при подключении отправляет регистрационное сообщение с уникальным логическим именем
- Брокер сохраняет соответствие между ZeroMQ identity клиента и его логическим именем
- При отключении клиента брокер помечает его как недоступного
- При повторном подключении клиента брокер восстанавливает сессию и доставляет накопленные сообщения

### 3.2. Отправка сообщений
- Клиент может отправить сообщение любому зарегистрированному клиенту по логическому имени
- Сообщение содержит:
  - Тип сообщения
  - Флаги (включая `NEEDS_REPLY`)
  - Correlation ID (уникальный идентификатор запроса)
  - Имя отправителя
  - Имя получателя
  - Полезную нагрузку (бинарные данные)

### 3.3. Двухсторонняя связь (Request-Reply)
- Клиент может отправить сообщение с флагом `NEEDS_REPLY`
- Брокер сохраняет связь между Correlation ID и отправителем
- Получатель обрабатывает сообщение и отправляет ответ с тем же Correlation ID
- Брокер доставляет ответ исходному отправителю (даже если он временно offline)

### 3.4. Гарантированная доставка
- Все входящие сообщения синхронно сохраняются в SQLite до подтверждения доставки
- Если получатель недоступен, сообщение сохраняется со статусом `pending`
- При подключении получателя все pending-сообщения автоматически доставляются
- После получения подтверждения (ACK) от клиента сообщение помечается как `delivered`
- При запуске брокера все pending-сообщения загружаются для последующей доставки

### 3.5. Управление сессиями
- Брокер отслеживает состояние всех подключённых клиентов
- При обрыве соединения клиент помечается как offline, но его очередь сохраняется
- Поддерживается множественное подключение клиентов

### 3.6. Протокол обмена
- Используется бинарный протокол поверх ZeroMQ
- Формат сообщения: фиксированный заголовок + переменные поля
- ZeroMQ обеспечивает фреймовую структуру и маршрутизацию по identity

## 4. Нефункциональные требования

### 4.1. Технологический стек
| Компонент | Технология |
|-----------|------------|
| Язык | C++17/20 |
| Сеть | Boost.Asio |
| Транспорт | ZeroMQ (libzmq + cppzmq) |
| Хранение | SQLite |
| Логирование | spdlog |
| Сборка | CMake |
| Тестирование | Google Test |

### 4.2. Требования к производительности
- Пропускная способность: не менее 10 000 сообщений/сек
- Задержка доставки онлайн-сообщений: < 1 мс (без учёта сети)
- Поддержка не менее 1000 одновременных подключений

### 4.3. Требования к надёжности
- Сообщения не теряются при падении брокера
- Корректное восстановление состояния после перезапуска
- Обработка обрывов соединения без утечек ресурсов

### 4.4. Требования к коду
- Модульная архитектура с чётким разделением ответственности
- Покрытие ключевых компонентов тестами
- Документирование публичного API

## 7. Порядок контроля и приёмки

### 7.1. Демонстрация базового сценария
1. Два клиента регистрируются под разными именами
2. Один клиент отправляет сообщение с запросом ответа
3. Второй клиент получает сообщение и отправляет ответ
4. Первый клиент получает ответ

### 7.2. Демонстрация отказоустойчивости
1. Отключение клиента-получателя
2. Отправка сообщения в офлайн
3. Переподключение клиента — сообщение доставляется
4. Перезапуск брокера — все неотправленные сообщения восстанавливаются
