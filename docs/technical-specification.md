# Техническое задание

## на разработку асинхронного брокера сообщений с гарантированной доставкой и персистентным хранилищем на основе библиотек Boost.Asio и ZeroMQ

---

## 1. Введение

### 1.1. Наименование работы
**«Асинхронный брокер сообщений с гарантированной доставкой и персистентным хранилищем на основе библиотек Boost.Asio и ZeroMQ»**

### 1.2. Краткое описание
Разрабатывается отдельный сетевой сервис (брокер), обеспечивающий надёжный асинхронный обмен сообщениями между распределёнными процессами. Брокер гарантирует доставку сообщений даже при временной недоступности получателя за счёт сохранения на диск, поддерживает двухстороннюю связь (request-reply) с корреляцией сообщений, подтверждение доставки (ACK) и предоставляет клиентам асинхронный интерфейс через ZeroMQ.

---

## 2. Назначение и цели разработки

### 2.1. Назначение
Обеспечение надёжного асинхронного обмена сообщениями между распределёнными компонентами (процессами, микросервисами) с гарантированной доставкой, поддержкой обратной связи и экспортом метрик для мониторинга.

### 2.2. Цели разработки
1. Реализовать брокер как самостоятельный процесс.
2. Обеспечить гарантированную доставку сообщений за счёт персистентного хранения на диске (SQLite) и механизма подтверждений (ACK).
3. Реализовать двухстороннюю связь с корреляцией запросов и ответов.
4. Обеспечить отложенную доставку сообщений для офлайн-получателей.
5. Реализовать асинхронную обработку всех операций на базе Boost.Asio, интегрировав ZeroMQ в цикл событий через файловый дескриптор (FD).
6. Предоставить клиентам API для регистрации, отправки сообщений, получения ответов и graceful shutdown (Unregister).
7. Реализовать экспорт метрик в формате Prometheus для мониторинга состояния брокера.

## 3. Функциональные требования

### 3.1. Регистрация клиентов
- Клиент при подключении отправляет регистрационное сообщение (`MessageType::Register`) с уникальным логическим именем.
- Брокер сохраняет соответствие между ZeroMQ identity клиента и его логическим именем.
- При повторном подключении клиента брокер восстанавливает сессию (вытесняя старую) и доставляет накопленные сообщения.

### 3.2. Отправка сообщений
- Клиент может отправить сообщение (`MessageType::Message`) любому зарегистрированному клиенту по логическому имени.
- Сообщение содержит:
    - Тип сообщения
    - Флаги (`NEEDS_REPLY`, `NEEDS_ACK`)
    - Correlation ID (уникальный идентификатор запроса)
    - Имя отправителя
    - Имя получателя
    - Полезную нагрузку (бинарные данные)

### 3.3. Двухсторонняя связь (Request-Reply)
- Клиент может отправить сообщение с флагом `NEEDS_REPLY`.
- Брокер сохраняет связь между Correlation ID и оригинальным отправителем в таблице `correlations`.
- Получатель обрабатывает сообщение и отправляет ответ (`MessageType::Reply`) с тем же Correlation ID.
- Брокер доставляет ответ исходному отправителю, используя сохраненную корреляцию (даже если отправитель временно offline).

### 3.4. Гарантированная доставка и подтверждения (ACK)
- Все входящие сообщения синхронно сохраняются в SQLite со статусом `STATUS_PENDING` (0).
- Если получатель недоступен, сообщение остается в статусе `STATUS_PENDING`.
- При отправке сообщения онлайн-клиенту с флагом `NEEDS_ACK` статус меняется на `STATUS_SENT` (2).
- При получении подтверждения (`MessageType::Ack`) от клиента сообщение помечается как `STATUS_DELIVERED` (1).
- Если ACK не получен в течение таймаута (`AckTimeout`), статус сообщения сбрасывается обратно в `STATUS_PENDING` для повторной попытки доставки.
- Сообщения без флага `NEEDS_ACK` помечаются как `STATUS_DELIVERED` сразу после успешной отправки сокету.

### 3.5. Управление сессиями
- Брокер отслеживает состояние всех подключённых клиентов (`is_online`).
- При обрыве соединения клиент помечается как offline, а его очередь исходящих сообщений персистируется в БД.
- Фоновый поток периодически (`SessionTimeout`) удаляет неактивные сессии.

### 3.6. Метрики (Prometheus)
- Брокер экспортирует метрики по HTTP на отдельном порту (`MetricsBindAddress`).
- **Счетчики (Counter)**:
    - `broker_messages_total`: received, sent, failed, ack, expired, offline.
    - `broker_clients_total`: register, unregister, timeout.
- **Датчики (Gauge)**:
    - `broker_state`: active_sessions, pending_send_queue.
- **Гистограммы (Histogram)**:
    - `broker_message_processing_duration_seconds`: время обработки одного сообщения.
    - `broker_payload_size_bytes`: размер полезной нагрузки сообщения.

### 3.7. Протокол обмена
- Используется бинарный протокол поверх ZeroMQ (ROUTER/DEALER).
- Формат сообщения: `[Версия (1б)] [Тип (1б)] [Флаги (1б)] [Correlation ID (8б)] [Длина Sender (1б)] [Длина Dest (1б)] [Длина Payload (2б)] [Sender] [Destination] [Payload]`.
- ZeroMQ обеспечивает фреймовую структуру и маршрутизацию по identity.

- Пропускная способность: не менее 10 000 сообщений/сек (на локальной петле).

- Сообщения не теряются при рестарте брокера (восстановление из БД).
- Корректная обработка отключения клиента (persist очереди сессии в БД).
- Корректная обработка просроченных ACK (retry логика).

### 4.4. Требования к коду
- Модульная архитектура с использованием интерфейсов (IStorage, IMessageSender, ISessionManager, IMetrics).

- Документирование публичного API.

## 7. Порядок контроля и приёмки

### 7.1. Демонстрация базового сценария
1. Два клиента регистрируются под разными именами.
2. Один клиент отправляет сообщение с запросом ответа (`NEEDS_REPLY` и `NEEDS_ACK`).
3. Второй клиент получает сообщение, отправляет `ACK`, затем отправляет `Reply`.
4. Первый клиент получает `ACK` и `Reply`.

### 7.2. Демонстрация отказоустойчивости
1. Отключение клиента-получателя.
2. Отправка сообщения в офлайн (сохраняется в БД со статусом `PENDING`).
3. Переподключение клиента — брокер автоматически отправляет накопленное сообщение.
4. Принудительная перезагрузка брокера.
5. После запуска брокер загружает неотправленные сообщения из БД и доставляет их подключившимся клиентам.
